# Аудит репозитория: согласованность, риски и точки улучшения

## Что проверено

- Логика запуска и потока данных (`main.py`).
- Парсинг карточек и fallback на браузерный рендер (`parser.py`).
- Формирование PDF и загрузка изображений (`pdf.py`).
- Работа с Google Sheets / Drive (`sheets.py`, `drive.py`, `auth.py`).
- Конфигурация и секреты (`settings.py`, `README.md`).

## Найденные риски и потенциальные ошибки

### 1) SSRF-риск из-за необработанных ссылок

Ранее скрипт принимал ссылки из таблицы без ограничений по хосту и схеме и мог отправлять запросы к внутренним адресам (например, `localhost`, `127.0.0.1`, приватные IP).

**Риск:** утечка данных из внутренней сети/инфраструктуры runner-а, нецелевой сетевой доступ.

**Что сделано:** добавлена валидация URL в `parser.py` и `pdf.py`:
- разрешены только `http/https`;
- заблокированы `localhost`, loopback, private/link-local/multicast/reserved IP.

### 2) Потенциальный рост памяти при инлайне изображений

Ранее `image_to_data_uri` загружал весь файл целиком в память (`response.content`), что на больших картинках могло приводить к повышенному расходу RAM и нестабильности pipeline.

**Риск:** OOM/замедление и нестабильная генерация PDF.

**Что сделано:** добавлена потоковая загрузка изображения с лимитом 8 MB. При превышении лимита функция возвращает исходный URL, а не data URI.

### 3) Излишне «тихие» fallback-и

В `image_to_data_uri` ошибки проглатываются и просто возвращается URL. Это полезно для устойчивости, но усложняет диагностику причин плохого PDF.

**Рекомендация:** добавить опциональный debug-лог (например, через `logging` + уровень `INFO/WARNING`) и фиксировать тип ошибки/домен.

### 4) Нет ограничений по доменам-источникам

Сейчас проект допускает парсинг любого публичного домена (после валидации безопасности URL).

**Рекомендация:** если бизнес-задача ограничена конкретными магазинами, стоит добавить allowlist доменов (через переменную окружения) для более предсказуемого качества и снижения случайных ошибок.

## Согласованность с заявленной задачей

В целом проект согласован с заявленной целью:
- берет ссылки из Sheets;
- заполняет поля парсинга;
- формирует PDF;
- опционально грузит PDF на Drive.

Критичных архитектурных расхождений не обнаружено. Основные улучшения — безопасность URL и эксплуатационная устойчивость при работе с изображениями.

## Варианты следующих улучшений

1. Добавить централизованный `logging` и флаг `--debug` для диагностики.
2. Добавить unit-тесты для:
   - `is_safe_url`/`is_safe_image_url`;
   - нормализации цены;
   - миграции листа в `normalize_values`.
3. Добавить ограничение на максимальный размер HTML/ответа при статическом парсинге.
4. В `run_summary.json` добавить отдельный блок `warnings` с причинами fallback-ов.
